# ERC-4337: Account Abstraction Standard

## Overview

ERC-4337 introduces **Smart Contract Wallets** that operate independently of Externally Owned Accounts (EOAs). It abstracts away the complexity of wallet mechanics, enabling better UX, security, and programmability.

**Key Innovation:** Anyone can create a smart contract wallet without ETH

---

## Core Concepts

### Traditional EOA Model

```
User → EOA (Private Key) → Transaction → Smart Contract
                ↑
         Must have ETH for gas
```

### Account Abstraction Model

```
User → Smart Contract Wallet → Mempool → Bundler → Network
             ↑
        Programmable logic
        Abstract gas mechanics
        Batch operations
        Social recovery
```

---

## Key Components

### 1. EntryPoint Contract
- Central contract coordinating user operations
- Validates and executes operations
- Handles gas accounting

### 2. User Operations (UserOps)
Transactions initiated by smart wallets:

```solidity
struct UserOperation {
    address sender;                    // Wallet address
    uint256 nonce;                     // Counter to prevent replay
    bytes initCode;                    // Factory & constructor data
    bytes callData;                    // Call to execute
    uint256 callGasLimit;              // Gas for execution
    uint256 verificationGasLimit;      // Gas for verification
    uint256 preVerificationGas;        // Pre-verification gas
    uint256 maxFeePerGas;              // Max fee per gas
    uint256 maxPriorityFeePerGas;      // Priority fee
    bytes paymasterAndData;            // Paymaster info
    bytes signature;                   // Wallet signature
}
```

### 3. Bundlers
- Collect user operations
- Bundle them into transactions
- Submit to entry point
- Receive rewards

## Key Interfaces

### IAccount

```solidity
interface IAccount {
    /**
     * Validate user's signature and nonce
     */
    function validateUserOp(
        UserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 missingAccountFunds
    ) external returns (uint256 validationData);
}
```

### IPaymaster (Optional)

```solidity
interface IPaymaster {
    /**
     * Payment validation - can sponsor gas
     */
    function validatePaymasterUserOp(
        UserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 requiredPreFunding
    ) external returns (bytes memory context, uint256 validationData);

    /**
     * Post-operation handler
     */
    function postOp(
        PostOpMode mode,
        bytes calldata context,
        uint256 actualGasCost
    ) external;
}
```

## Implementation Patterns

### Pattern 1: Basic Smart Wallet

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import "@account-abstraction/contracts/core/BaseAccount.sol";

contract SimpleWallet is BaseAccount {
    address public owner;
    uint256 public nonce = 0;
    
    event OwnerInitialized(address indexed owner);
    
    constructor(address _owner) {
        owner = _owner;
    }
    
    function validateUserOp(
        UserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 missingAccountFunds
    ) public override returns (uint256 validationData) {
        // Verify owner signature
        bytes memory signature = userOp.signature;
        address signer = recoverSigner(userOpHash, signature);
        
        if (signer != owner) {
            return SIG_VALIDATION_FAILED;
        }
        
        return 0;
    }
    
    function execute(
        address dest,
        uint256 value,
        bytes calldata func
    ) public {
        require(msg.sender == address(entryPoint()) || msg.sender == owner);
        
        (bool success,) = dest.call{value: value}(func);
        require(success);
    }
    
    function recoverSigner(
        bytes32 hash,
        bytes memory signature
    ) internal pure returns (address) {
        // Use ECDSA recovery
    }
}
```

### Pattern 2: Multi-Sig Wallet

```solidity
contract MultiSigWallet is BaseAccount {
    address[] public owners;
    uint256 public requiredSignatures;
    
    mapping(bytes32 => mapping(address => bool)) public confirmations;
    mapping(bytes32 => bool) public executed;
    
    function validateUserOp(
        UserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 missingAccountFunds
    ) public override returns (uint256 validationData) {
        // Verify multi-sig approval
        uint256 confirmationCount = 0;
        
        for (uint256 i = 0; i < owners.length; i++) {
            if (confirmations[userOpHash][owners[i]]) {
                confirmationCount++;
            }
        }
        
        require(confirmationCount >= requiredSignatures, "Not enough confirmations");
        
        return 0;
    }
}
```

## Gas Optimization

1. **Batch Operations**
```javascript
// Traditional
await send(tx1);  // Pay gas 3 times
await send(tx2);
await send(tx3);

await bundler.send([tx1, tx2, tx3]);  // Pay gas once
```


## Paymaster Pattern: Sponsored Transactions

```solidity
contract GaslessDappPaymaster is IPaymaster {
    address public entryPoint;
    address public dapp;
    
    function validatePaymasterUserOp(
        UserOperation calldata userOp,
        bytes32 userOpHash,
        uint256 requiredPreFunding
    ) external override returns (bytes memory context, uint256 validationData) {
        // Check if operation is for our dapp
        require(isDappOperation(userOp), "Not dapp operation");
        
        // Paymaster will cover gas
        return ("", 0);
    }
    
    function postOp(
        PostOpMode mode,
        bytes calldata context,
        uint256 actualGasCost
    ) external override {
        // Charge dapp for gas used
    }
}
```

## Wallet Factory Pattern

```solidity
contract WalletFactory {
    event WalletCreated(address indexed wallet, address indexed owner);
    
    function createWallet(address owner, uint256 salt)
        external
        returns (address)
    {
        SimpleWallet wallet = new SimpleWallet{salt: bytes32(salt)}(owner);
        emit WalletCreated(address(wallet), owner);
        return address(wallet);
    }
    
    function getWalletAddress(address owner, uint256 salt)
        public
        view
        returns (address)
    {
        return address(
            new SimpleWallet{salt: bytes32(salt)}(owner)
        );
    }
}
```

## Advantages

1. **Better UX** - No ETH for gas initially
2. **Batch Operations** - Multiple actions, one transaction
3. **Programmability** - Logic in wallets, not just keys
4. **Recovery** - Multiple recovery methods
5. **Standards** - Cross-protocol compatibility
6. **Sponsored Gas** - Dapps can cover costs

---

## Disadvantages

1. **Complexity** - Requires bundlers, infrastructure
2. **Adoption** - Wallets need support
3. **Latency** - Bundle time adds delay
4. **Costs** - More processing (offset by batching)
5. **Immuturity** - Still evolving standard

---

## Security Considerations

1. **Signature Verification** - Use battle-tested libraries
2. **Nonce Management** - Prevent replay attacks
3. **Paymaster Trust** - Verify paymaster reliability
4. **Guardian Management** - Secure guardian selection
5. **Upgrade Safety** - Careful proxy management
