# ERC-721: Non-Fungible Token Standard

## Overview

ERC-721 is the foundational standard for NFTs on Ethereum. It defines a set of functions and events that allow for the creation and management of non-fungible tokens, where each token is unique and has a distinct identity.

**Key Principle:** 1 NFT = 1 unique token

---

## Core Concepts

### Token Ownership
- Each token has a unique ID
- Each token is owned by exactly one address at any time
- Tokens cannot be divided

### Transfer Mechanics
- Only the owner or approved address can transfer a token
- Transfers are permanent and immutable
- Can be approved for a single transfer or multiple transfers

---

## Key Functions

### Query Functions
```solidity
// Get the owner of a specific token
function ownerOf(uint256 tokenId) external view returns (address owner);

// Get total number of tokens owned by an address
function balanceOf(address owner) external view returns (uint256);

// Get the total supply of tokens
function totalSupply() external view returns (uint256);

// Get the approved address for a specific token
function getApproved(uint256 tokenId) external view returns (address);

// Check if an address is approved to operate all tokens for another
function isApprovedForAll(address owner, address operator) external view returns (bool);
```

### Mutation Functions
```solidity
// Transfer token from owner to recipient
function transferFrom(address from, address to, uint256 tokenId) external;

// Transfer token with data callback (used for smart contracts)
function safeTransferFrom(address from, address to, uint256 tokenId, bytes calldata data) external;

// Approve an address to transfer a specific token
function approve(address to, uint256 tokenId) external;

// Approve/revoke an address to manage all tokens for caller
function setApprovalForAll(address operator, bool approved) external;
```

### Metadata Functions
```solidity
// Get the name of the token collection
function name() external view returns (string memory);

// Get the symbol/ticker
function symbol() external view returns (string memory);

// Get metadata URI (usually points to JSON file)
function tokenURI(uint256 tokenId) external view returns (string memory);
```

---

## Events

```solidity
// Emitted when a token is transferred
event Transfer(address indexed from, address indexed to, uint256 indexed tokenId);

// Emitted when an address is approved for a specific token
event Approval(address indexed owner, address indexed approved, uint256 indexed tokenId);

// Emitted when approval is granted for all tokens
event ApprovalForAll(address indexed owner, address indexed operator, bool approved);
```

## Best Use Cases

✅ **Perfect For:**
- Digital art and collectibles
- Certificates and credentials
- Domain names (ENS)
- Virtual real estate
- Gaming assets with unique properties
- Membership tokens

❌ **Not Ideal For:**
- Fungible tokens (use ERC-20 instead)
- Large collections of identical items (use ERC-1155)
- Highly variable supply scenarios

---

## Advantages

1. **Simplicity** - Easy to understand and implement
2. **Wide Adoption** - Most wallets and platforms support it
3. **Proven Security** - Well-audited and battle-tested
4. **Metadata Support** - Native URI support for off-chain data
5. **Standard Approval Flow** - Familiar transfer patterns

---

## Disadvantages

1. **Single Token Type** - Each contract = one type of NFT
2. **Gas Inefficiency** - Separate storage for each token
3. **Limited Batch Operations** - No native batch transfer
4. **Metadata Hosting** - Usually stored off-chain (centralization risk)

## Implementation Considerations

### 1. Minting Strategy
- Sequential IDs (0, 1, 2, ...)
- Random IDs
- Custom ID schemes

### 2. Access Control
- Only contract owner can mint
- Allow public minting
- Role-based access

### 3. Metadata Storage
- Centralized (AWS, server)
- Decentralized (IPFS, Arweave)
- On-chain (expensive but permanent)

### 4. Burning
- Allow users to burn their tokens
- Reduce supply over time
- Implement carefully

## Common Patterns

### Ownable Minting
```solidity
// Only contract owner can mint
modifier onlyOwner() {
    require(msg.sender == owner);
    _;
}

function mint(address to, uint256 tokenId) external onlyOwner {
    // Minting logic
}
```

### Pausable Transfers
```solidity
// Can pause/unpause transfers in emergencies
function transferFrom(address from, address to, uint256 tokenId) external {
    require(!paused, "Transfers paused");
    // Transfer logic
}
```

### Whitelist Minting
```solidity
// Only whitelisted addresses can mint
mapping(address => bool) public whitelist;

function mint(uint256 tokenId) external {
    require(whitelist[msg.sender], "Not whitelisted");
    // Minting logic
}
```

## Security Considerations

1. **Re-entrancy** - Use `nonReentrant` qualifier on transfers
2. **Amount Validation** - Validate token IDs
3. **Approval Dos** - Prevent approval bypasses
4. **Metadata Integrity** - Ensure metadata URI validity
5. **Overflow/Underflow** - Use SafeMath (Solidity 0.8+ has built-in checks)

## Migration Consideration

If you need multiple token types in one contract, consider **ERC-1155** instead.

If you need rental mechanics, layer **ERC-4907**.

For royalties, implement **ERC-2981**.
