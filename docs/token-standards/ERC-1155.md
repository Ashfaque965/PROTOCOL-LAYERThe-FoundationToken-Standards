# ERC-1155: Multi-Token Standard

## Overview

ERC-1155 is a revolutionary standard that supports both fungible tokens (like currencies) and non-fungible tokens (like unique items) within a **single contract**. It's the standard of choice for gaming and complex token ecosystems.

**Key Innovation:** One contract for all token types

---

## Core Concepts

### Multiple Token Types
- Single contract can have unlimited token types
- Each token ID represents either:
  - A fungible token (balance per address)
  - A non-fungible token (owned by one address)
  - A hybrid (fungible with special instances)

### Unified Transfer Mechanism
- Batch transfers supported natively
- Single approval for all token types
- Efficient storage and gas

### Balance Model
```
balanceOf: mapping(address => mapping(uint256 => uint256))
// For NFT: balance = 1 (owned) or 0 (not owned)
// For FT: balance = any amount
```

## Key Functions

### Query Functions
```solidity
// Get balance of a specific token for an address
function balanceOf(address account, uint256 id) external view returns (uint256);

// Batch get balances
function balanceOfBatch(
    address[] calldata accounts,
    uint256[] calldata ids
) external view returns (uint256[] memory);

// Check if operator is approved for all tokens
function isApprovedForAll(address account, address operator) external view returns (bool);

// Get URI for token metadata
function uri(uint256 id) external view returns (string memory);
```

### Mutation Functions
```solidity
// Transfer single token
function safeTransferFrom(
    address from,
    address to,
    uint256 id,
    uint256 amount,
    bytes calldata data
) external;

// Batch transfer multiple tokens
function safeBatchTransferFrom(
    address from,
    address to,
    uint256[] calldata ids,
    uint256[] calldata amounts,
    bytes calldata data
) external;

// Grant operator approval for all tokens
function setApprovalForAll(address operator, bool approved) external;
```

## Best Use Cases

✅ **Perfect For:**
- Gaming ecosystems (weapons, armor, currency, land)
- Trading card games
- Mixed economies (currency + items)
- Fractional NFTs
- Inventory systems
- Staking rewards distribution
- Batch operations

❌ **Not Ideal For:**
- Simple single NFT collections (ERC-721 is simpler)
- Fungible-only tokens (ERC-20 is lighter)
- Tokens needing per-type metadata (use ERC-721)

---

## Advantages

1. **Gas Efficiency** - Batch operations reduce costs dramatically
2. **Flexibility** - Support fungible + non-fungible in one contract
3. **Scalability** - Unlimited token types
4. **Batch Operations** - Transfer multiple items in one transaction
5. **Atomic Transfers** - Either all succeed or all fail
6. **Metadata Privacy** - Can update metadata without contract upgrade

---

## Disadvantages

1. **Complexity** - More complex than ERC-721
2. **Support Variability** - Not all wallets support it equally
3. **Learning Curve** - Harder for beginners
4. **ID Management** - Developer must manage ID schemes

## Token Type Examples

### Example ID Scheme

```
ID Format: [category_bits][item_bits]

Category (bits 0-7):     ID 0-255
  - 0-99: Fungible tokens
  - 100-199: Weapons
  - 200-299: Armor
  - etc.

Item-specific (bits 8-255)
```

### Example Mapping

```solidity
uint256 constant GOLD_TOKEN = 1;          // Fungible
uint256 constant SILVER_TOKEN = 2;        // Fungible
uint256 constant LEGENDARY_SWORD = 101;   // Non-fungible
uint256 constant COMMON_SHIELD = 201;     // Non-fungible
```

## Metadata Standard

### URI Strategy

```
// Option 1: Individual URIs per token
function uri(uint256 id) public view override returns (string memory) {
    return string(abi.encodePacked(baseURI, id));
}

// Option 2: Dynamic URI
function uri(uint256 id) public view override returns (string memory) {
    uint256 category = id >> 8;
    return string(abi.encodePacked(baseURI, category));
}
```

### JSON Format
```json
{
  "name": "Item Name",
  "description": "Item Description",

  "properties": {
    "category": "weapon",
    "rarity": "legendary",
    "damage": 50
  }
}
```

## Implementation Patterns

### 1. Gaming Ecosystem

```solidity
// Fungible tokens
uint256 constant GOLD = 1;
uint256 constant MANA = 2;

// Non-fungible items
uint256 constant SWORD = 100;
uint256 constant ARMOR = 101;

// Mint currency
function mintGold(address to, uint256 amount) external {
    _mint(to, GOLD, amount, "");
}

// Mint unique item
function mintSword(address to) external {
    _mint(to, SWORD, 1, "");
}
```

### 2. Batch Operations

```solidity
function equip(uint256[] memory items) external {
    uint256[] memory amounts = new uint256[](items.length);
    for (uint i = 0; i < items.length; i++) {
        amounts[i] = 1;
    }
    
    _safeBatchTransferFrom(msg.sender, equipmentVault, items, amounts, "");
}
```

### 3. Mixed Transfers

```solidity
function tradeItems(
    address recipient,
    uint256[] memory sends,
    uint256[] memory sendAmounts,
    uint256[] memory receives,
    uint256[] memory receiveAmounts
) external {
    // Send offer
    _safeBatchTransferFrom(msg.sender, recipient, sends, sendAmounts, "");
    
    // Receive counter-offer
    _safeBatchTransferFrom(recipient, msg.sender, receives, receiveAmounts, "");
}
```

## ID Management Best Practices

### 1. Non-Overlapping Ranges
```solidity
// Reserve ranges for different categories
uint256 constant FUNGIBLE_START = 1;
uint256 constant FUNGIBLE_END = 100;

uint256 constant WEAPONS_START = 1000;
uint256 constant WEAPONS_END = 2000;

uint256 constant ARMOR_START = 2000;
uint256 constant ARMOR_END = 3000;
```

### 2. Bit Packing
```solidity
// Pack information in ID
function getId(uint8 category, uint32 itemId) public pure returns (uint256) {
    return (uint256(category) << 32) | itemId;
}

function parseId(uint256 id) public pure returns (uint8 category, uint32 itemId) {
    category = uint8(id >> 32);
    itemId = uint32(id);
}
```

## Receiver Contract Hook

When transferring to contracts, they must implement `IERC1155Receiver`:

```solidity
interface IERC1155Receiver {
    function onERC1155Received(
        address operator,
        address from,
        uint256 id,
        uint256 value,
        bytes calldata data

    function onERC1155BatchReceived(
        address operator,
        address from,
        uint256[] calldata ids,
        uint256[] calldata values,
```

## Comparison: ERC-721 vs ERC-1155

| Feature | ERC-721 | ERC-1155 |
|---------|---------|----------|
| Token Types | Single per contract | Multiple per contract |
| Batch Operations | No | Yes |
| Fungible Support | No | Yes |
| Gas Cost | Higher for multiple | Lower |
| Simplicity | Higher | Lower |
| Learning Curve | Easier | Harder |
